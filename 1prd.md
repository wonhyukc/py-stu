# Project Requirements Document (1prd.md)

## 1. 프로젝트 목적 (Purpose)
- `eval`은 학생들의 성적과 과제를 자동화하여 처리하는 파이썬 기반의 관리 시스템입니다.
- **SSOT (Single Source of Truth)**: 이 문서는 프로젝트의 전역 설정, 아키텍처 및 전반적인 규칙을 정의하는 유일한 기준 문서입니다.
- **성적 평가 기준**: 전체 학기 성적 산출 비중(기말 40%, 중간 20%, 과제 20%, 참여도 20%)에 대한 공식 규정은 [`docs/grading-policy.md`](./docs/grading-policy.md) 문서를 따릅니다.

## 2. 주요 기능 및 연동 (Core Features & Integrations)
1. **Google Sheets API 연동**:
   - 학생 성적 및 과제 제출 현황 데이터를 읽고 쓰기 위한 메인 데이터베이스 역할.
   - 대상 시트 링크:
     - **파이썬 과정**: [파이썬 과제 시트](https://docs.google.com/spreadsheets/d/1F69Wtmrr3MYMJI8jgjPKBWh3m8tDaX6QyBVDSHH4vEc/edit?gid=1892167835#gid=1892167835)
     - **웹 과정**: [웹 과제 시트](https://docs.google.com/spreadsheets/d/1XA5Hnu5PEidFMreanPCy1eMrofGiyFDPo4buN3129Mc/edit?gid=1514293361#gid=1514293361)
     - **과제 제출 및 상호평가 통합 폼**: [통합 폼 시트](https://docs.google.com/spreadsheets/d/1_o-F6UaQ2WOe0nH2zuT_0xpwiOm1ebmWo2sEQzQptEk/edit?resourcekey=&gid=427896056#gid=427896056&fvid=8905371)
   - 인증 방식: 서비스 계정(Service Account, `secret.json`)을 활용.
2. **이메일 수신 및 성적 처리 (과제 채점 자동화)**:
   - **수신 대상 계정**: 읽어올 대상 메일 계정은 반드시 `wonhyukc@stu.ac.kr` 이어야 합니다.
   - **수신 및 분석 (Gmail 필터 규칙)**: 실행 시 파라미터(예: 파이썬은 `과제 0.4`, 웹은 `assignment 0.4`)로 메일 검색 쿼리를 입력받습니다. 
     - **과정별 키워드 분리**: 파이썬 과정은 제목에 반드시 **'과제'**가, 웹 과정은 **'assignment'**가 포함되어야 구분하여 각각의 시트에 채점이 가능합니다.
     - 넘겨진 쿼리값은 `settings.json`에 저장되어 다음 실행 시 디폴트 값으로 사용됩니다.
     - 사용자가 파라미터를 입력하지 않으면 `settings.json`의 기존(디폴트) 쿼리 값을 그대로 사용하여 타겟 이메일을 수집합니다.
     - **본인 메일 제외**: 스크립트 실행자 및 교강사 본인의 이메일(`wonhyukc@stu.ac.kr`) 발송분은 처리 대상에서 제외합니다.
     - **학번 추출**: 이메일 제목에 기재된 8~11자리(예: 10자리) 숫자를 파싱하여 학번으로 식별하고, 메일 제목이 긴 경우 20자까지만 자릅니다.
   - **자동 채점 및 결과 저장 로직**:
     - **배정 한계점**: 상호평가 매칭 시, 해당 분반의 총 제출자 수가 10명 이하일 경우 동일한 피평가자가 중복 배정될 수 있는 문제점이 있으며, 이는 스크립트 실행 직후의 자동 채점표 점검 로직에 의해 터미널에 사전 경고됩니다.
     - **채점 기준**: 마감 시간 내에 도착한 이메일은 무조건 '1점'을 부여합니다.
     - **중복 제출 방어**: 동일한 학번으로 메일이 여러 번(예: 오류 수정, 쓰레드 피드백 등) 수신된 경우, 가장 최신 메일 1건에 대해서만 1점을 부여하고 나머지 중복 데이터는 제외합니다.
     - **CSV 결과 출력**: 파싱된 데이터는 `학번, 점수, 이유, 날짜, 이름, 메일제목` 규격의 CSV 포맷으로 1차 변환됩니다.
     - **구글 시트 기록 방식 (Append)**: 기존 시트에 작성되어 있는 데이터 행은 어떤 경우에도 덮어쓰지 않으며, 반드시 마지막 행 아래의 새로운 빈 행부터 순차적으로 데이터를 추가(Append) 기록합니다.
3. **인증 및 보안 (Credentials)**:
   - `secret.json`: 프로젝트 루트에 위치하며, Google Cloud/Workspace API 접근을 위한 서비스 계정 키를 보관합니다. 절대 Git 저장소에 커밋되지 않도록 `.gitignore` 등으로 관리해야 합니다.

## 3. 구조 및 모듈 작성 규칙 (Architecture & Module Rules)
- **가상환경 필수 (Virtual Environment)**: 파이썬 패키지 의존성 관리 및 스크립트 실행 시 로컬 가상환경(`.venv` 등)을 활성화하여 사용해야 합니다.
- **모듈(`modules/`) 생성 규칙**:
  - 각 기능(예: 시트 처리, 이메일 발송 등)은 독립된 모듈로 분리하여 작성합니다.
  - 모듈 내에는 자신만의 단위 테스트 파일과 최소한의 로컬 문서를 두되, 문서 다이어트(AI-MD-Diet) 원칙에 따라 잡다한 로그보다는 **'현재 상태 요약(What/Why)'** 위주로만 남깁니다.
- **실행 스크립트**:
  - `bin/` 또는 메인 디렉토리의 엔트리포인트를 통해 실행 가능하게 구성합니다.
- **테스트**:
  - `pytest` 등의 프레임워크를 기반으로 각 기능별 단위 테스트 및 E2E 테스트를 작성합니다.

## 4. 커밋 및 언어 정책 (Commit & Language Policy)
- **언어**: 모든 문서와 코드 주석, 커밋 메시지는 한국어(한글 전용)로 작성합니다.
- **커밋**: 사용자의 명시적인 명령이 없는 한 로컬 커밋까지만 진행하며, `git push`는 수행하지 않습니다.
  - 관련 작업 완료 시 커밋 전 반드시 Github 이슈를 생성해야 합니다.
  - 커밋 메시지(제목 또는 본문)에 해당 이슈 번호를 포함해야 합니다.
  - 기존 `3history.md`를 대체하여, 커밋 메시지 본문(Body)에 작업 맥락과 상세 이력을 상세하게 기록해야 합니다. (전사 워크플로우 `/commit-clean` 규칙 준수)

## 5. AI 도구 경계 및 제약 (AI Tool Boundaries & Constraints)
- **파일 접근 제약**: AI 에이전트는 `secret.json` 및 `credentials.json` 등 인증 관련 파일에 절대로 접근(읽기/쓰기)해서는 안 됩니다.
- **명령어 제약**: DB 초기화나 중요 파일을 삭제하는 위험한 OS 명령어(`rm -rf` 등) 사용은 금지됩니다. (안전모드 실행)
- **단일 소스 진실 (SSOT)**: 모든 가이드라인은 본 `1prd.md`를 우선하며, 작업은 `.agent-rules`나 하네스 테스트를 무조건 통과해야 합니다.



### **4. Data Format Rule**
- **CSV 전용 (TSV 사용 금지)**: 모든 표 형태의 데이터 출력 및 파일 생성은 예외 없이 쉼표(`,`)로 구분된 `CSV(Comma-Separated Values)` 포맷만을 사용합니다. 탭(`TSV`) 포맷은 지원 스펙이 아니므로 어떤 상황에서도 생성하거나 제안하지 않습니다.
- **결과 파일명 라우팅 (Filename Routing)**: 하위 폴더(서브 폴더)를 절대로 생성하지 않으며 모든 추출 결과는 `output/` 디렉토리 최상단에 저장합니다. 단, 터미널 매개변수가 주어졌을 경우 `mail07_py.csv`와 같이 파일명에 해당 매개변수(주차, 트랙)를 덧붙여 구분하여 출력합니다.
