# 학생 과제 제출 및 상호 평가 시스템 도입 계획 (eval-plan: GitHub 강제화 및 단계별 도입 모델)

**문서 목적**: 학생들의 원활한 GitHub 적응과 자동 채점 체계(기한 검열, 상호 평가 합산)를 결합한 최적의 아키텍처 및 확정된 운영 전략 명시.

---

## 1. 핵심 요구사항 (GitHub 기반)

1. **GitHub 사용 강제**: 학생들은 무조건 GitHub을 통해 코드를 관리하고 과제를 제출해야 함.
2. **마감 전 비공개, 마감 후 전체 공유**: 최종적으로는 기한 전 훔쳐보기를 차단하되, 직후에는 모두가 서로의 코드를 열람(상호 평가)할 수 있어야 함.
3. **엄격하고 조작 불가능한 채점 자동화**: 기한 내 제출 여부 및 코드 무단 수정 여부를 엄격히 판별하여 기본 점수를 자동 부여.
4. **상호 평가 점수의 자동 합산 및 투명화**: 수동 평가는 학생이 하지만, 결과 합산과 메인 구글 시트 동기화는 자동화되어야 함.

---

## 2. 제안하는 GitHub 혼합형 워크플로우 (Git Hash + Google Forms)

GitHub의 특성상 Public/Private 권한 문제와 커밋 시간 조작 리스크를 해결하기 위해, **`GitHub Commit Hash 기반 제출 구조`**를 근간으로 합니다.

### 📍 [1단계: 과제 제출]
- 학생들은 자신의 과제 Repository를 생성하여 작업합니다.
- 과제를 완성하면 가장 최신 버전의 **`Commit Hash(영문/숫자 고유값)`**와 **`Repository Link`**를 **'과제 제출용 Google Form'**을 통해 제출합니다.
- **검증 보장**: Google Form은 서버에서 타임스탬프를 엄격하게 기록하므로, 제출된 `Commit Hash`는 "마감 시간 이전에 명백히 존재했다"는 증명이 됩니다.

### 📍 [2단계: eval 스크립트를 통한 1차 자동 채점]
- 파이썬 `eval` 시스템이 '과제 제출용 Google Form'을 읽어옵니다.
- **GitHub API 연동**: 각 학생의 공개된(Public) Repo 링크를 순회하며, 학생들이 제출했던 **`Commit Hash` 값이 해당 Repo에 존재하는지 검증**합니다.
  - ✔️ 해당 Commit이 존재함: 기한 내 제출 완료 (기본 점수).
  - ❌ 해당 Commit이 없거나 코드를 덮어씀: 과제 소스 조작으로 간주하여 점수 미부여.

### 📍 [3단계: 상호 평가 (Peer Review)] 상세 프로세스
**3-1. 평가 대상자 시트 자동 추출**  
파이썬 `eval` 스크립트가 1차 검증(기한 내 Commit 제출 여부)을 통과한 학생들의 데이터만 추려냅니다. 추출된 데이터(학번, 이름, 과제 GitHub 링크)는 전체 학생에게 읽기 전용으로 공유되는 **'상호 평가 대상자 구글 시트'**에 자동으로 업데이트됩니다.
*(주의: 과제를 기한 내에 제출하지 못한 '미제출자'라 하더라도, 동료의 코드를 열람하고 채점하는 상호 평가에는 의무적으로 참여해야 합니다. 만약 채점에 불참할 경우 과제 제출 성적과는 별개로 '상호 평가 참여 점수(예: 1.0 체제 내 참여분)'를 전혀 부여받지 못합니다.)*

**3-2. 동료 과제 코드 열람 및 분석**  
과제 제출자 및 미제출자를 포함한 100% 전체 학생은 공유된 대상자 시트에 접속하여, **본인에게 랜덤 배당된 3명의 동기 과제**를 채점 대상자로 확인하고 해당 GitHub 링크로 이동합니다. 코드를 직접 열람하고 구조를 파악하며 요구사항에 맞춰 테스트를 진행합니다.

**3-3. 상호 평가 폼 제출 및 범용 매트릭스 활용 (0/1 바이너리 채점)**  
코드 분석을 마친 학생은 **'상호 평가용 Google Form'**을 통해 리뷰를 제출합니다. 
루브릭은 Pass(1점) 아니면 Fail(0점)로 판정되는 아주 단순한 기준들로 구성됩니다. 매 주차별로 폼을 새로 생성하는 번거로움을 피하기 위해 구글 폼의 문항은 범용적인 **[평가기준 1], [평가기준 2]** 형식으로 고정하며, 평가자는 각 기준에 대해 0점 또는 1점만을 부여합니다. 
*(매 주차 채점 점수 총합은 항상 최대 1.0 만점의 소수점으로 환산/계산됩니다.)*
* **제출 항목**: [본인 학번(Reviewer)], [대상자 학번(Reviewee)], [기준별 부여 점수(0점 또는 1점)], [정성적 피드백(Code Review)]

**3-4. 채점자 신뢰성 검증 및 페널티 부여 (어뷰징 방지)**  
단순 0/1 매트릭스 환경에서 무성의하거나 악의적인 채점자(Outlier)를 필터링해야 합니다. 한 명의 과제를 3명이 0/1로 한정되게 평가하므로, **특정 채점자가 다른 2명과 명확히 상반된(사실과 다른) 판정을 내릴 경우 "신뢰할 수 없는 채점"으로 간주하여 페널티를 부여**합니다.

* **[기본 원칙: 다수결 배제 알고리즘]**: 단일 문항(0 or 1)에 대해 3명 중 다수(2명 이상)의 의견을 정답(Ground Truth)으로 취급하여 즉시 점수를 확정합니다. 혼자 다수결과 엇갈린 판정을 한 소수 채점자는 해당 기준에서 '신뢰도 경고'가 1회 누적되고, 기계적으로 페널티(평가 참여 점수 박탈 등)가 부여됩니다.
* **[예외 처리: 이의 제기(Appeal) 프로세스]**: 위 다수결 원칙에 의해 페널티를 받은 소수 채점자가 "내 채점이 맞고 나머지 2명이 틀렸다"며 이의를 제기할 경우에 한정하여 예외 프로세스가 발동됩니다. 해당 케이스는 즉각 별도의 시트(`Review Required` 플래그)로 이관되며, 교강사가 직접 양쪽 평가와 소스 코드를 실사하여 어느 리뷰어가 옳았는지 정성적으로 최종 판결을 내립니다.



### 📍 [4단계: 최종 성적 자동 합산 및 분기] 상세 프로세스
**4-1. 다중 시트 데이터 병합 (Data Aggregation)**  
교강사가 실행하는 `eval` 병합 스크립트가 1) 원본 '과제 제출 폼 시트'(기본 시간 내 제출 점수)와 2) '상호 평가 폼 응답 시트'(동료들의 평가 점수 및 피드백 모음)의 데이터를 동시에 읽어와 학번을 Key값으로 하여 하나로 병합(Join)합니다.

**4-2. 과정별 타겟 마스터 시트 분기 (Routing)**  
채점 봇은 실행 시 입력된 파라미터 쿼리를 확인합니다. 메일 제목 검색 쿼리가 파이썬 키워드(`과제 0.x`)인지 웹 키워드(`assignment 0.x`)인지에 따라, `1prd.md`에 정의된 **파이썬 전용 성적 시트** 또는 **웹 전용 성적 시트**로 향할 타겟 URL을 자동으로 분기 확정합니다.

**4-3. 최종 점수 연산 및 Append (투명화 완성)**  
[기본 제출 점수 + 상호 평가 평균 점수] 기반으로 최종 점수가 산식 처리됩니다. 최종 결과(학번, 이름, 최종 점수, 서술 피드백 요약)는 타겟으로 지정된 과정별 메인 성적 관리 시트의 최하단 행에 **순차적으로 Append (추가 기록)** 됨으로써, 과거 데이터 조작 없이 모든 학생들의 평가 과정을 투명하게 기록합니다.

## 3. 🎯 [최종 확정] 매 주차 과제 운영: 단계별 도입 전략 (Phased Rollout)

학생들의 Git 숙련도 부족으로 인한 초기 혼란을 방지하기 위해, 학기 진행 수준에 맞춘 **'단계적 옵션 적용 방식'**을 최종 확정하여 운영합니다.

### 🟢 1단계: 초기 적응기 (전면 Public Repository 운영)
학습 극초반, 학생들이 Git과 GitHub의 기본 개념(Commit, Push, Repository 생성 등)조차 생소할 때 적용하는 임시 운영 방식.
* **운영 플로우**: 학생들은 매 주차 과제별로 **처음부터 아예 Public Repo를 생성**하여 작업하고 폼으로 제출합니다.
* **기대 효과**: 'Private에서 Public으로 가시성을 변경해야 한다'는 과정 자체를 생략하여 학생들의 인지 부하와 제출 오류(404 Error)를 완전히 덜어냅니다. 
* **감수할 리스크**: 마감 전에 다른 학생의 과제를 훔쳐볼 수 있는 약점이 있습니다. 하지만 초창기엔 "코드의 완벽함"보다 "Git으로 폼까지 제출해보는 경험" 자체가 훨씬 중요하므로, 일정 기간 이 리스크를 감수합니다.

### 🔵 2단계: 안정화기 (매 주차 옵션 A: Private 생성 → Public 전환)
학생들이 Commit Hash를 폼으로 제출하고 과제를 운영하는 생태계에 완전히 익숙해졌을 때(예: 중간고사 직전이나 일정 주차 이후) 도입하는 최종 완성형 방식.
* **운영 플로우 (옵션 A)**: 
  1. 학생들은 매 주차 과제를 **새로운 Private Repo(`web-week0N`)**로 생성하여 철저히 비공개로 작업합니다.
  2. 마감 전, Private 상태에서 폼으로 커밋 해시를 제출합니다.
  3. 마감 직후, 학생 본인이 직접 레포지토리 설정(Settings)에 들어가 **가시성(Visibility)을 Public으로 영구 전환**합니다.
* **기대 효과**: 초기에 감수했던 '마감 전 코드 유출' 문제를 완벽히 틀어막으면서도, 시스템적으로는 여전히 커밋 해시 검증이 100% 동작하므로 가장 이상적인 '상호 평가 환경'이 확립됩니다.
