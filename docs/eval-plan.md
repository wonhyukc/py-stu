# 학생 과제 제출 및 상호 평가 시스템 도입 계획 (eval-plan: GitHub 강제화 및 단계별 도입 모델)

**문서 목적**: 학생들의 원활한 GitHub 적응과 자동 채점 체계(기한 검열, 상호 평가 합산)를 결합한 최적의 아키텍처 및 확정된 운영 전략 명시.

---

## 1. 핵심 요구사항 (GitHub 기반)

1. **GitHub 사용 강제**: 학생들은 무조건 GitHub을 통해 코드를 관리하고 과제를 제출해야 함.
2. **마감 전 비공개, 마감 후 전체 공유**: 최종적으로는 기한 전 훔쳐보기를 차단하되, 직후에는 모두가 서로의 코드를 열람(상호 평가)할 수 있어야 함.
3. **엄격하고 조작 불가능한 채점 자동화**: 기한 내 제출 여부 및 코드 무단 수정 여부를 엄격히 판별하여 기본 점수를 자동 부여.
4. **상호 평가 점수의 자동 합산 및 투명화**: 수동 평가는 학생이 하지만, 결과 합산과 메인 구글 시트 동기화는 자동화되어야 함.

---

## 2. 제안하는 GitHub 혼합형 워크플로우 (Git Hash + Google Forms)

GitHub의 특성상 Public/Private 권한 문제와 커밋 시간 조작 리스크를 해결하기 위해, **`GitHub Commit Hash 기반 제출 구조`**를 근간으로 합니다.

### 📍 [1단계: 과제 제출]
- 학생들은 자신의 과제 Repository를 생성하여 작업합니다.
- 과제를 완성하면 가장 최신 버전의 **`Commit Hash(영문/숫자 고유값)`**와 **`Repository Link`**를 **'과제 제출용 Google Form'**을 통해 제출합니다.
- **검증 보장**: Google Form은 서버에서 타임스탬프를 엄격하게 기록하므로, 제출된 `Commit Hash`는 "마감 시간 이전에 명백히 존재했다"는 증명이 됩니다.

### 📍 [2단계: eval 스크립트를 통한 1차 자동 채점]
- 파이썬 `eval` 시스템이 '과제 제출용 Google Form'을 읽어옵니다.
- **GitHub API 연동**: 각 학생의 공개된(Public) Repo 링크를 순회하며, 학생들이 제출했던 **`Commit Hash` 값이 해당 Repo에 존재하는지 검증**합니다.
  - ✔️ 해당 Commit이 존재함: 기한 내 제출 완료 (기본 점수).
  - ❌ 해당 Commit이 없거나 코드를 덮어씀: 과제 소스 조작으로 간주하여 점수 미부여.

### 📍 [3단계: 상호 평가 (Peer Review)]
- 검증에 통과한 학생들의 GitHub 링크가 **'전체 상호 평가 대상자 시트(공유용)'**에 로직으로 자동 출력됩니다.
- 학생들은 이 시트에서 동기들의 코드를 열람한 뒤, **'상호 평가용 Google Form'**에 평가 점수와 피드백을 기재합니다.

### 📍 [4단계: 최종 평가점수 자동 합산]
- 파이썬 `eval` 스크립트가 [기본 제출 점수]와 [상호 평가 폼 점수]를 병합/연산합니다.
- 최종 결과를 `1prd.md`에 정의된 **eval 관리 시트 (성적 마스터 DB)** 하단에 **Append (추가 기록)** 하여 전체 프로세스를 완성합니다.

## 3. 🎯 [최종 확정] 매 주차 과제 운영: 단계별 도입 전략 (Phased Rollout)

학생들의 Git 숙련도 부족으로 인한 초기 혼란을 방지하기 위해, 학기 진행 수준에 맞춘 **'단계적 옵션 적용 방식'**을 최종 확정하여 운영합니다.

### 🟢 1단계: 초기 적응기 (전면 Public Repository 운영)
학습 극초반, 학생들이 Git과 GitHub의 기본 개념(Commit, Push, Repository 생성 등)조차 생소할 때 적용하는 임시 운영 방식.
* **운영 플로우**: 학생들은 매 주차 과제별로 **처음부터 아예 Public Repo를 생성**하여 작업하고 폼으로 제출합니다.
* **기대 효과**: 'Private에서 Public으로 가시성을 변경해야 한다'는 과정 자체를 생략하여 학생들의 인지 부하와 제출 오류(404 Error)를 완전히 덜어냅니다. 
* **감수할 리스크**: 마감 전에 다른 학생의 과제를 훔쳐볼 수 있는 약점이 있습니다. 하지만 초창기엔 "코드의 완벽함"보다 "Git으로 폼까지 제출해보는 경험" 자체가 훨씬 중요하므로, 일정 기간 이 리스크를 감수합니다.

### 🔵 2단계: 안정화기 (매 주차 옵션 A: Private 생성 → Public 전환)
학생들이 Commit Hash를 폼으로 제출하고 과제를 운영하는 생태계에 완전히 익숙해졌을 때(예: 중간고사 직전이나 일정 주차 이후) 도입하는 최종 완성형 방식.
* **운영 플로우 (옵션 A)**: 
  1. 학생들은 매 주차 과제를 **새로운 Private Repo(`web-week0N`)**로 생성하여 철저히 비공개로 작업합니다.
  2. 마감 전, Private 상태에서 폼으로 커밋 해시를 제출합니다.
  3. 마감 직후, 학생 본인이 직접 레포지토리 설정(Settings)에 들어가 **가시성(Visibility)을 Public으로 영구 전환**합니다.
* **기대 효과**: 초기에 감수했던 '마감 전 코드 유출' 문제를 완벽히 틀어막으면서도, 시스템적으로는 여전히 커밋 해시 검증이 100% 동작하므로 가장 이상적인 '상호 평가 환경'이 확립됩니다.
